## 1. 프로젝트 개요

### 1.1 배경

기존 중고거래 플랫폼은 상품 등록과 단순 채팅 기능에 집중되어 있어, **직거래 약속 잡기**, **예약 거래**, **신뢰도 기반 거래 보호** 등 실제 거래 과정에서 필요한 기능이 분리되거나 부재한 경우가 많다. 본 플랫폼은 상품 등록부터 예약, 채팅 협상, 거래 완료, 리뷰까지 **하나의 서비스 안에서 end-to-end 거래 경험**을 제공한다.

### 1.2 서비스 정의

**Tradev**는 개인 간(C2C) 중고 물품 거래 및 서비스 예약을 지원하는 웹 플랫폼이다.  
사용자는 물품을 판매하거나 구매할 수 있으며, 판매자가 지정한 시간 슬롯에 예약을 잡아 직거래를 진행할 수 있다. Claude API 기반 AI 기능으로 상품 설명 자동완성, 가격 추천, 사기 패턴 감지 등을 제공한다.

### 1.3 핵심 가치

| 가치 | 설명 |
|------|------|
| **신뢰** | 거래 상태 머신, 리뷰/평가, 신고 시스템으로 안전한 거래 환경 제공 |
| **편의** | AI 자동완성으로 상품 등록 시간 단축, 예약 캘린더로 직거래 약속 간소화 |
| **투명성** | 거래 이력, 리뷰, 신뢰 점수를 공개해 사용자 판단 근거 제공 |

## 2. 목표 및 성공 지표

### 2.1 비즈니스 목표

- 상품 등록부터 거래 완료까지 **단일 플랫폼**에서 처리 가능한 서비스 구축
- 예약 시스템 도입으로 **노쇼(No-show) 최소화**
- AI 기능으로 **상품 등록 UX 개선** 및 **거래 신뢰도 향상**

### 2.2 포트폴리오 목표

- Docker + GitHub Actions 기반 **실제 배포 운영 경험** 확보
- 거래 상태 머신, 예약 슬롯 충돌 방지 등 **실무 도메인 로직** 구현
- Claude API 연동 **AI-assisted 기능** 포함
- Spring Boot + Vue.js 풀스택 + 인프라까지 **end-to-end 포트폴리오** 완성

### 2.3 성공 지표 (KPI)

| 지표 | 목표값 |
|------|--------|
| 상품 등록 후 AI 자동완성 사용률 | ≥ 60% |
| 예약 후 거래 완료 전환율 | ≥ 70% |
| 신고 처리 후 재발 방지율 | ≥ 80% |
| 서버 응답 시간 (p95) | ≤ 500ms |
| CI/CD 빌드 → 배포 소요 시간 | ≤ 5분 |

## 3. 사용자 정의

### 3.1 사용자 유형

#### 일반 사용자 (User)
- 회원 가입 후 플랫폼 이용
- 판매자(Seller)와 구매자(Buyer) 역할을 동시에 수행 가능
- 상품 등록, 구매, 예약, 채팅, 리뷰 작성 가능

#### 관리자 (Admin)
- 내부 관리자 계정 (별도 가입 불가, 시드 데이터로 생성)
- 회원 관리, 상품 관리, 신고 처리, 통계 조회 가능
- 전용 관리자 페이지(MPA) 접근

### 3.2 페르소나

**페르소나 A — 대학생 박지훈 (판매자)**  
사용하지 않는 교재, 전자기기를 팔고 싶다. 상품 설명 쓰기가 번거롭고 적정 가격을 모른다.  
→ AI 설명 자동완성, 가격 추천 기능이 필요.

**페르소나 B — 직장인 이소연 (구매자)**  
믿을 수 있는 판매자에게 중고 물품을 사고 싶다. 약속 시간을 잡는 게 귀찮다.  
→ 신뢰 점수, 리뷰 확인 기능 + 예약 캘린더 기능이 필요.

### 4.3 거래 시스템

#### 4.3.1 거래 상태 머신

```
[판매중] ──구매 요청──▶ [요청 대기]
                              │
              ┌───── 판매자 수락 ──────┐
              ▼                        ▼
         [예약중]               [거래 거절됨] ──▶ (종료)
              │
    거래 완료 확인 (양측)
              │
              ▼
         [거래완료]
              │
        ┌─────┴─────┐
        ▼           ▼
   [리뷰 작성]  [리뷰 작성]
   (구매자)     (판매자)

[판매중] / [예약중] ──취소──▶ [취소됨] ──▶ (종료)
```

| 상태 | 코드 | 설명 |
|------|------|------|
| 판매중 | `SALE` | 구매 가능한 상태 |
| 요청 대기 | `PENDING` | 구매 요청 후 판매자 수락 대기 |
| 예약중 | `RESERVED` | 판매자가 수락, 직거래 예약 또는 택배 진행 중 |
| 거래완료 | `COMPLETED` | 양측 확인 후 완료 |
| 취소됨 | `CANCELLED` | 거래 도중 취소 |
| 거절됨 | `REJECTED` | 판매자가 거부 |

#### 4.3.2 거래 요청 흐름

| ID | 기능 | 우선순위 | 설명 |
|----|------|----------|------|
| TRADE-01 | 구매 요청 | 🔴 | 구매자가 상품에 구매 요청, 메시지 첨부 가능 |
| TRADE-02 | 요청 수락 / 거절 | 🔴 | 판매자가 요청 목록에서 처리 |

## 5. 비기능 요구사항

### 5.1 성능

| 항목 | 목표 |
|------|------|
| API 응답 시간 (p95) | ≤ 500ms |
| 상품 목록 조회 (p99) | ≤ 1,000ms |
| 채팅 메시지 전송 지연 | ≤ 100ms |
| 이미지 업로드 최대 크기 | 파일당 5MB, 상품당 최대 10장 |
| 동시 WebSocket 연결 수 | ≥ 200 (단일 인스턴스 기준) |

### 5.2 보안

- JWT Access Token 만료 시간: **30분**
- Refresh Token 만료 시간: **7일** (Redis 저장, Rotation 전략)
- 비밀번호: **BCrypt** 해싱
- API 요청 Rate Limit: IP 기준 **100 req/min**
- SQL Injection, XSS 방지 (Spring Security + 입력 검증)
- CORS: 허용 Origin 명시 (프론트엔드 도메인만)
- HTTPS 적용 (Nginx SSL Termination)

### 5.3 가용성 / 배포

- 목표 업타임: **99% 이상** (개인 프로젝트 기준)
- 무중단 배포: Docker Compose Blue-Green 전략 (단순화 버전)
- 헬스체크 엔드포인트: `GET /actuator/health`
- 에러 발생 시 Slack 알림 (🟢 Nice to Have)

### 5.4 확장성

- 서비스 레이어 단위 분리로 추후 MSA 전환 고려
- 이미지 스토리지는 AWS S3 (로컬 개발 시 MinIO 대체 가능)
- DB 커넥션 풀: HikariCP 기본 설정 활용

### 5.5 로깅 / 모니터링

- 애플리케이션 로그: Logback (JSON 포맷, 레벨별 분리)
- API 요청/응답 로깅: Spring AOP 또는 Filter
- 에러 트래킹: 로그 파일 + Docker 로그 드라이버
- DB 슬로우 쿼리 로깅: ≥ 1,000ms

## 7. 제약사항 및 가정

### 7.1 제약사항

- 결제 시스템은 **포함하지 않음** (추후 확장 예정)
- 모바일 앱 미제공 (반응형 웹으로 대응)
- 이메일 발송 서비스: **AWS SES** 또는 **SMTP** (가입 인증, 비밀번호 재설정용)
- 이미지 처리(리사이징)는 클라이언트 사전 처리 또는 S3 + Lambda 추후 적용
- 다국어(i18n) 미지원, 한국어 단일 언어

### 7.2 가정

- 단일 EC2 인스턴스 운영 (수평 확장 미적용)
- 사용자 규모: 동시 접속 ≤ 500명 (포트폴리오 기준)
- 슬롯 임시 잠금 TTL: **5분**
- AI 기능은 비동기 처리, 응답 지연 시 UX 저하 없도록 스트리밍 또는 로딩 처리

## 9. 용어 정의

| 용어 | 정의 |
|------|------|
| **슬롯 (Slot)** | 판매자가 직거래를 위해 등록하는 날짜/시간 단위 |
| **슬롯 임시 잠금** | 구매자가 예약 요청 시 다른 사람이 동일 슬롯을 요청하지 못하도록 TTL 기반으로 잠그는 것 |
| **거래 상태 머신** | 상품이 판매중 → 요청대기 → 예약중 → 거래완료로 전이되는 상태 관리 체계 |
| **신뢰 점수** | 거래 완료, 리뷰, 신고 제재에 따라 변동되는 사용자 신뢰 지표 |
| **SSE** | Server-Sent Events, 서버에서 클라이언트로 단방향 실시간 푸시 기술 |
| **낙관적 잠금** | 데이터 충돌을 version 컬럼으로 감지하여 동시 수정 방지하는 전략 |
| **MPA** | Multi-Page Application, 관리자 페이지에 사용하는 전통적인 서버 렌더링 방식 |
| **SPA** | Single-Page Application, Vue.js 기반 사용자 페이지 방식 |
